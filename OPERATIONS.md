# OPERATIONS.md — 研究運用の正典

このファイルは、AIエージェントと人間が共有する**研究プロジェクトの運用仕様**である。
特定のツール・モデルに依存しない。ここに書かれた「何をすべきか」だけが正であり、
「誰が・どの環境で実行するか」は各実行環境のアダプタ側の関心事とする。

> 規律: このファイルには、特定の実行環境だけに適用される条件分岐や呼び出し方法を書かない。
> 固有の配置場所・コマンド名・機能制約はアダプタに閉じ込める。

---

## 0. 全体像 — 3層アーキテクチャ

```
ログ層(追記専用)     活動ログ・原記録・git — 機械的な事実。最強の正。書き換え禁止
   ↓ 蒸留
文書層(完全版)      共有docs/と登録済みルート。網羅性が正義、鮮度は二の次。原則write-once
   ↓ 蒸留
STATE/ 層(文脈パック) 行動に必要な要約 + docsへのポインタ。鮮度が正義、網羅は捨てる
```

- **STATE と文書層で「正義」が逆**。STATEは古い情報が残ることが罪、文書層は情報が欠けることが罪。
- 矛盾を見つけたときの信頼順位は **ログ > 文書層 > STATE**(下の層ほど書き換えられておらず信頼できる)。
- 同じ詳細は **ちょうど1箇所** に住む。登録済み文書に全文、STATEには要点1行とリンクを置く。

---

## 1. STATE/ 層 — 文脈パック

STATE/ だけを読めば、初見の協力者(人間でもコンテクストゼロのエージェントでも)が
研究の立ち位置を掴んで作業に入れる状態を目指す。**変化速度で分ける**。

| ファイル | 目的 | 変化速度 |
|---|---|---|
| `README.md` | 目次 + 最小STATE(現在の問い/最も強い結果と根拠/次の一手/直近やったこと) | 速い(毎セッション) |
| `motivation.md` | なぜこの研究か、誰が困っているか、何ができたら成功か(反証可能な成功条件) | ほぼ静的 |
| `related-work.md` | 既存研究の地図。要約でなく「我々との差分/借りるもの/なぜそれでは解けないか」 | ほぼ静的 |
| `design.md` | 研究課題・アプローチ・評価方法・制約・**捨てた選択肢とその理由** | ゆっくり |
| `materials.md` | 対象・資料・データ・試料・参加者など、研究材料の要点と妥当性上の注意 | ゆっくり |
| `decisions.md` | 決定ログ。**追記のみ**。人間が承認した方向転換やパネルの採否を落とす | 追記 |
| `<topic>.md` | features / leakage-checks など。**トピック単位**で切る(日付単位にしない) | 速い |

`README.md` の冒頭には、同期境界を曖昧にしないため次の機械可読なカーソルを置く。

```yaml
---
last_sync_at: YYYY-MM-DDTHH:MM:SS+09:00
last_sync_commit: <git commit SHA>
activity_log_cursors: {}
---
```

- `last_sync_at` はタイムゾーン付きISO 8601。日付だけにしない。
- `last_sync_commit` は前回SYNCで確認済みのコミット。初回は空文字列。
- `activity_log_cursors` は `research-ops.yml` に登録した活動ログごとの最終確認位置。
  JSONLなら行数または安定した記録ID、追記専用CSVなら行数など、再開点を一意に表す値を使う。

規約:
- 各ファイルの冒頭に**目的文**を置く(例:「初見の協力者が一読で立ち位置を掴むためのもの。
  流し読みできなくなったら目的を失っている」)。**行数などの数値制約は設けない**。
- README は目次だけにしない。最小STATEを兼ね、README しか読まれなくても致命傷にならないようにする。
- 静的・低速なファイルは冒頭に `最終確認: YYYY-MM-DD` を持つ。
- 存在しない任意ファイルを README の目次に先回りして載せない。ファイルを追加したSYNCで目次も追加する。

## 2. 文書層 — 完全版

STATE が参照する、細部まで抜けのない記録。
文書層は研究全体で一つだが、物理的な保存場所は複数でもよい。
リポジトリ直下の `docs/` を共有文書の既定位置とし、サブテーマ固有の結果や考察は
`individual/`、`population/` のような既存フォルダに残せる。

すべての保存場所を `research-ops.yml` の `document_roots` に登録する。

```yaml
document_roots:
  - id: shared
    path: docs
    scope: project
    include: "**/*.md"
  - id: individual
    path: individual
    scope: individual-prediction
    include: "**/*.md"
  - id: population
    path: population
    scope: population-prediction
    include: "**/*.md"
```

- `id` は同期結果と参照で使う一意な名前。
- `path` はリポジトリルートからの相対パス。ディレクトリと単一ファイルのどちらも指定できる。
- `scope` は記録が扱う研究全体またはサブテーマの名前。
- `include` はディレクトリ内で文書として扱うglob。省略時は `**/*.md`。
  `path` が単一ファイルなら使わない。
- 同じファイルが複数のルートに一致する重複登録は避ける。
- 同じ詳細は一つの文書だけに置く。共有文書にはサブテーマ固有文書への要約とリンクを置く。

共有文書の推奨構成は次のとおり。

```
docs/
├── activities/    # 実験・観察・分析・調査など、研究活動ごとの詳細記録
├── sources/       # 論文・史料・インタビューなど、参照資料ごとの完全なノート
├── materials/     # データ・試料・コーパス・対象システムなどの完全な説明
├── methods/       # 手法・手順・分析方法・実装上の注意
├── explorations/  # 捨てた方向の検討記録の完全版(design.md の「捨てた理由」の裏付け)
├── reviews/       # PANELなどの審議記録
└── notes/         # 議論・アイデアのダンプ(未整理でよい ← 摩擦最小の受け口)
```

規約:
- **書き込みは文書層ファースト**。実験、観察、文献調査、検討を**したその瞬間**に詳細を書く
  (あとから思い出して書く詳細は必ず欠ける)。
- **原則 write-once**。書いたレポート・ノートは後から書き換えない(継続更新が要るのは materials/ と methods/ 程度)。
- ただし**覆された結論には errata の一行追記を義務とする**(書き換えず打ち消す)。
  例: 「※この結論は decisions.md 2026-08-01 で棄却(前提条件の誤りが判明)」。
- `notes/` は殴り書きでよい。整った記録を求めて記録自体をサボるより、雑でも残すことを優先する。

## 3. 蒸留 — 文書層 → STATE(一方向)

STATE を最新化する作業。逆流(STATEの詳細を根拠なしに文書層へ戻すこと)はしない。

- **差分駆動**: 前回同期以降に変わった登録済み文書とログだけを処理対象にする。変わっていないSTATEには触らない。
- **自己申告禁止**: 「今日やった気がすること」ではなく、git log、活動ログ、原記録から逆算して書く。
- **削除も仕事の一部**: もう行動に影響しない項目は、トピックファイルに沈殿させるか消す。
  消すのは情報の喪失ではない。**完全版は文書層にある**。残すなら理由を一行付ける。
  (明示しないとエージェントは足す一方で消さず、STATE が単調膨張して要約でなくなる。)
- **載せる/落とすの物差し**:
  > これを読んだ読み手が、よい研究を進める上で **今** 知っておくべきことか。
  > 迷ったら逆から検算する: これを知らないまま作業に入ると、判断や行動を誤りうるか。
  > 誤らないなら根拠文書へのリンクで十分。
- 目次腐敗(README の一行説明と各ファイルの実態のズレ)を毎回チェックする。

### 3.1 活動ログの発見

文書ルートと、実験、観察、分析、調査などの活動ログは、リポジトリ直下の `research-ops.yml` に明示する。
推測でログを探したり、未登録の結果を正として採用したりしない。

```yaml
version: 1
profile: general
document_roots:
  - id: shared
    path: docs
    scope: project
    include: "**/*.md"
activity_logs:
  - id: main
    kind: experiment
    path: activities/log.jsonl
    format: jsonl
    id_field: record_id
    timestamp_field: recorded_at
```

- `profile` は分野別の語彙・STATE雛形・PANEL役割を選ぶ名前。
- `document_roots` は完全版の文書を探す場所。共有文書とサブテーマ固有文書を一つの文書層として扱う。
- `id` は `activity_log_cursors` のキーとして使う安定した名前。
- `kind` は `experiment` / `observation` / `analysis` / `source-review` など、記録の種類を表す任意の名前。
- `path` はリポジトリルートからの相対パス。
- `format` は少なくとも `jsonl` / `csv` / `markdown` のいずれか。
- 活動ログがまだ無い場合は `activity_logs: []` とし、SYNCはその事実を警告する。
- ログを追加・移動したら、設定とカーソルを同じ変更で更新する。

## 4. 導入は最小から(重要)

**最初から全フォルダを建てない。** プロセスの維持が研究を食う(研究活動より STATE の手入れに時間を使う)
のが最大のリスク。初期ほど方向が週単位で変わり、丁寧に作った STATE はすぐ陳腐化する。

- 開始時に作るのは **`STATE/README.md` + `STATE/decisions.md` + `docs/README.md` + `research-ops.yml` + 既存の活動ログへの登録**。
  活動ログがまだ無ければ空の設定で開始し、最初の記録と同時に登録する。
- `design.md` は最初の方向転換のときに、`related-work.md` は既存研究を整理するときに、
  `docs/` の各サブフォルダは**実際にそこへ書くものが出たとき**に生まれればよい。
- 体系の劣化・膨張は、ルールで縛らず **AUDIT(§7)の実測**で検査する。

## 5. 呼び出しの階段

呼び出しの正体は常に「以下の手続き(§6-8)を、選んだAIに実行させる」の一つ。
自動化の度合いが違うだけ:

- **階段0(常に効く下限)**: 手続きの指示文をAIに手で渡す。ファイルアクセスの無い素のチャットなら対象ファイルも貼る。
- **階段1(シェル)**: 標準入力でプロンプトを受けるCLIに手続きを流し込む。実行モデルはCLIを差し替えて切替。
- **階段2(ツールネイティブ)**: 各実行環境のコマンドまたはスキル機能に登録し、名前を指定して起動。
- **階段3(無人)**: git フックや cron から階段1を叩く(セッション終了時同期・日次照合)。

## 6. 手続き: SYNC(同期)

1. `STATE/README.md` の `last_sync_at` / `last_sync_commit` / `activity_log_cursors` と、
   `research-ops.yml` の文書ルートとログ定義を読む。欠落・不正な設定は推測で補わず報告する。
   初回の空カーソルは不正値ではない。初回は現在存在するdocsと登録済みログをすべて確認し、
   git履歴の開始点は人間が指定した基準コミット、指定がなければ到達可能な全履歴とする。
2. ground truth を収集する:
   - `last_sync_commit..HEAD` の `git log`(メッセージと変更ファイル)
   - 現在の未コミット差分(`git diff` と未追跡ファイル)。SYNC自身が生成したSTATEの差分は入力から除く
   - 各カーソルより後の活動ログ
   - 上記に含まれる、登録済み文書ルートの変更
3. §3 の規約で照合し、STATEの変更案を作る(差分駆動・削除含む・物差しで取捨・目次腐敗チェック)。
4. `decisions.md` は追記のみ。既存部分の書き換え・削除をしない。
5. **既定はdry-run** とし、変更案をdiff形式で報告して人間の承認を待つ。
   明示的に適用を指示された場合だけファイルを更新する。
6. 適用が成功した場合に限り、確認済みの `HEAD`、現在時刻、各ログの最終確認位置で
   READMEのカーソルを更新する。未コミット差分を取り込んだ場合は、その対象ファイルも報告する。

---

## 7. 手続き: AUDIT(監査 — 月次目安)

「STATE/ だけで文脈がほぼ伝わる」を**実測**する。

1. 現在の会話から隔離された実行者(新規セッションまたは独立コンテクスト)に、**STATE/ だけ**を読ませる。
   同じ会話内で「初見のつもり」になる代替は禁止する。隔離実行が利用できなければ監査を停止し、
   STATEだけを渡した新規セッションでの実行手順を報告する。
2. 次の3つを答えさせる:
   - このプロジェクトを説明せよ(目的・現在地・アプローチ)。
   - 次にやるべきことと、その理由を述べよ。
   - STATE/ だけでは分からなかったこと・確信が持てなかったことを列挙せよ。
3. 返ってきた**誤解**と**「分からなかった」リスト**が、そのまま STATE の欠落箇所。
4. あわせて機械チェック: STATE内リンク切れ、STATEの要約と登録済み文書の食い違い、
   文書ルート間の重複、`最終確認` が古すぎる静的ファイル。
5. 検出された欠落についてSYNCのdry-runを行い、修正案を人間へ提示する。

## 8. 手続き: PANEL(方向性の多視点審議)

`STATE/README.md`、選択中の研究プロファイル、必要最小限のdocsをブリーフに、方向性の妥当性を審議する。
素朴なマルチエージェント議論は追従的同調で「演劇」に収束するため、以下を守る:

- **独立ドラフト → 相互反論 の順**。最初からライブ会話にしない(会話は数ターンで同意に収束する)。
  各役割にまず他者の意見を見せずに独立メモを書かせ、次に「他者のメモを渡すので、
  同意できない点だけ具体的に反論せよ」の第2ラウンドを行い、最後に統合する。
- **役割は人格でなく「失敗モード」に紐づける**。
  具体的な役割は研究プロファイルで定めるが、意義、証拠の妥当性、代替説明、利用・影響、撤退条件を含める。
- **証拠への接地を必須にする**: 各主張に根拠(活動ID・資料・記録・コード箇所など)を付け、
  かつ「自分の主張が覆る条件と、それを確かめる活動を1つ」を添えさせる。
  これで出力が「感想」から「次にやるべき検証リスト」に変わる。
- **できれば実行者を異種にする**: 役割の一部を別ベンダーのモデルに割る。
  同一モデルから複数サンプリングしても意見分布は均質化する(既知の現象)ため。
- **パネル結果はまず `docs/reviews/panel-YYYYMMDD-<topic>.md` に保存する**。
  これは審議記録であり、正式な研究判断ではない。
- 人間が結論を承認した場合だけ、承認者・パネル記録へのリンク・採否を
  `STATE/decisions.md` に追記する。却下または保留もパネル記録側に残す。
