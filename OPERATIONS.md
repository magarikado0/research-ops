# OPERATIONS.md — 研究運用の正典

このファイルは、AIエージェントと人間が共有する**研究プロジェクトの運用仕様**である。
特定のツール・モデルに依存しない。ここに書かれた「何をすべきか」だけが正であり、
「誰が・どの環境で実行するか」はアダプタ(`claude-code/`, `codex/`)側の関心事とする。

> 規律: このファイルには特定のツール名・モデル名を書かない。
> 「Claude では」「Codex では」という分岐が一文でも混ざると移植性が崩れる。

---

## 0. 全体像 — 3層アーキテクチャ

```
ログ層(追記専用)     実験ログ・git — 機械的な事実。最強の正。書き換え禁止
   ↓ 蒸留
docs/ 層(完全版)     細部まで抜けのない記録。網羅性が正義、鮮度は二の次。原則write-once
   ↓ 蒸留
STATE/ 層(文脈パック) 行動に必要な要約 + docsへのポインタ。鮮度が正義、網羅は捨てる
```

- **STATE と docs で「正義」が逆**。STATE は古い情報が残ることが罪、docs は情報が欠けることが罪。
- 矛盾を見つけたときの信頼順位は **ログ > docs > STATE**(下の層ほど書き換えられておらず信頼できる)。
- 同じ詳細は **ちょうど1箇所** に住む。docs に全文、STATE には要点1行 + リンク。

---

## 1. STATE/ 層 — 文脈パック

STATE/ だけを読めば、初見の協力者(人間でもコンテクストゼロのエージェントでも)が
研究の立ち位置を掴んで作業に入れる状態を目指す。**変化速度で分ける**。

| ファイル | 目的 | 変化速度 |
|---|---|---|
| `README.md` | 目次 + 最小STATE(現在の仮説/ベストと実験ID/次の一手/直近やったこと) | 速い(毎セッション) |
| `motivation.md` | なぜこの研究か、誰が困っているか、何ができたら成功か(反証可能な成功条件) | ほぼ静的 |
| `related-work.md` | 既存研究の地図。要約でなく「我々との差分/借りるもの/なぜそれでは解けないか」 | ほぼ静的 |
| `design.md` | 問題の定式化・ターゲット定義・評価プロトコル・制約・**捨てた選択肢とその理由** | ゆっくり |
| `data.md` | データの説明・既知のクセ・リーク危険箇所 | ゆっくり |
| `decisions.md` | 決定ログ。**追記のみ**。方向転換やパネルの結論を落とす | 追記 |
| `<topic>.md` | features / leakage-checks など。**トピック単位**で切る(日付単位にしない) | 速い |

規約:
- 各ファイルの冒頭に**目的文**を置く(例:「初見の協力者が一読で立ち位置を掴むためのもの。
  流し読みできなくなったら目的を失っている」)。**行数などの数値制約は設けない**。
- README は目次だけにしない。最小STATEを兼ね、README しか読まれなくても致命傷にならないようにする。
- 静的・低速なファイルは冒頭に `最終確認: YYYY-MM-DD` を持つ。

## 2. docs/ 層 — 完全版

STATE が参照する、細部まで抜けのない記録。

```
docs/
├── data/          # データ辞書(全カラムの意味・型・欠損・出所)、収集/前処理の完全手順
├── experiments/   # 実験IDごとの詳細レポート(設定全文・結果・観察・考察)
├── papers/        # 論文ごとの読書ノート完全版(STATE/related-work.md はこの蒸留)
├── methods/       # 手法の詳細・数式・実装上の注意
├── explorations/  # 捨てた方向の検討記録の完全版(design.md の「捨てた理由」の裏付け)
└── notes/         # 議論・アイデアのダンプ(未整理でよい ← 摩擦最小の受け口)
```

規約:
- **書き込みは docs ファースト**。実験・論文読み・検討を**したその瞬間**に詳細を書く
  (あとから思い出して書く詳細は必ず欠ける)。
- **原則 write-once**。書いたレポート・ノートは後から書き換えない(編集が要るのは data/ と methods/ 程度)。
- ただし**覆された結論には errata の一行追記を義務とする**(書き換えず打ち消す)。
  例: 「※この結論は decisions.md 2026-08-01 で棄却(リーク判明)」。
- `notes/` は殴り書きでよい。整った記録を求めて記録自体をサボるより、雑でも残すことを優先する。

---

## 3. 蒸留 — docs → STATE(一方向)

STATE を最新化する作業。逆流(STATE の詳細を docs に戻す)はしない。

- **差分駆動**: 前回同期以降に変わった docs / ログだけを処理対象にする。変わっていない STATE には触らない。
- **自己申告禁止**: 「今日やった気がすること」ではなく、git log と実験ログから逆算して書く。
- **削除も仕事の一部**: もう行動に影響しない項目は、トピックファイルに沈殿させるか消す。
  消すのは情報の喪失ではない — **完全版は docs にある**。keep するなら理由を一行付ける。
  (明示しないとエージェントは足す一方で消さず、STATE が単調膨張して要約でなくなる。)
- **載せる/落とすの物差し**:
  > これを読んだ読み手が、よい研究を進める上で **今** 知っておくべきことか。
  > 迷ったら逆から検算する: これを知らないまま作業に入ると、判断や行動を誤りうるか。
  > 誤らないなら docs へのリンクで十分。
- 目次腐敗(README の一行説明と各ファイルの実態のズレ)を毎回チェックする。

---

## 4. 導入は最小から(重要)

**最初から全フォルダを建てない。** プロセスの維持が研究を食う(実験より STATE の手入れに時間を使う)
のが最大のリスク。初期ほど方向が週単位で変わり、丁寧に作った STATE はすぐ陳腐化する。

- 開始時に作るのは **`STATE/README.md` + `STATE/decisions.md` + 実験ログ** の3点だけ。
- `design.md` は最初の方向転換のときに、`related-work.md` は最初に論文を読み込んだときに、
  `docs/` の各サブフォルダは**実際にそこへ書くものが出たとき**に生まれればよい。
- 体系の劣化・膨張は、ルールで縛らず **AUDIT(§7)の実測**で検査する。

---

## 5. 呼び出しの階段

呼び出しの正体は常に「以下の手続き(§6-8)を、選んだAIに実行させる」の一つ。
自動化の度合いが違うだけ:

- **階段0(常に効く下限)**: 手続きの指示文をAIに手で渡す。ファイルアクセスの無い素のチャットなら対象ファイルも貼る。
- **階段1(シェル)**: 標準入力でプロンプトを受けるCLIに手続きを流し込む。実行モデルはCLIを差し替えて切替。
- **階段2(ツールネイティブ)**: 各ツールのコマンド/スキル機能に登録し、`/state-sync` 等で起動。
- **階段3(無人)**: git フックや cron から階段1を叩く(セッション終了時同期・日次照合)。

## 6. 手続き: SYNC(同期)

1. `STATE/README.md` を読み、最終同期日を確認する。
2. ground truth を収集する: 最終同期日以降の `git log`(メッセージと変更ファイル)、
   実験ログの新規行、`docs/` の変更ファイル。
3. §3 の規約で照合・更新する(差分駆動・削除含む・物差しで取捨・目次腐敗チェック)。
4. `decisions.md` は追記のみ。書き換え・削除をしない。
5. 変更した内容を diff 形式の要約で報告する。
   (導入初期は自動適用せず、この diff を人間が承認してから適用する。判断のズレはこの正典に追記して固定する。)

---

## 7. 手続き: AUDIT(監査 — 月次目安)

「STATE/ だけで文脈がほぼ伝わる」を**実測**する。

1. コンテクストを持たない実行者(できれば普段の蒸留とは別系統のモデル)に、**STATE/ だけ**を読ませる。
2. 次の3つを答えさせる:
   - このプロジェクトを説明せよ(目的・現在地・アプローチ)。
   - 次にやるべきことと、その理由を述べよ。
   - STATE/ だけでは分からなかったこと・確信が持てなかったことを列挙せよ。
3. 返ってきた**誤解**と**「分からなかった」リスト**が、そのまま STATE の欠落箇所。
4. あわせて機械チェック: STATE 内リンク切れ、STATE の要約と docs 実体の食い違い、`最終確認` が古すぎる静的ファイル。
5. 検出された欠落を SYNC で補う。

---

## 8. 手続き: PANEL(方向性の多視点審議)

`STATE/README.md`(必要なら docs)をブリーフに、方向性の妥当性を審議する。
素朴なマルチエージェント議論は追従的同調で「演劇」に収束するため、以下を守る:

- **独立ドラフト → 相互反論 の順**。最初からライブ会話にしない(会話は数ターンで同意に収束する)。
  各役割にまず他者の意見を見せずに独立メモを書かせ、次に「他者のメモを渡すので、
  同意できない点だけ具体的に反論せよ」の第2ラウンドを行い、最後に統合する。
- **役割は人格でなく「失敗モード」に紐づける**:
  - 査読者2号 — この予測タスクに新規性・意義があるか。既存研究で解決済みでは?
  - データ懐疑派 — リーク・ラベル品質・分布シフト。「そのスコア、本番でも出るのか」
  - ベースライン原理主義者 — 単純手法(線形・直近値・ルール)で十分では?MLの上乗せは本当にあるか
  - 出口担当 — その予測は誰が何に使うのか。評価指標は用途と一致しているか
  - 撤退判断人 — 「このプロジェクトを止めるべき理由」を全力で構築する
- **証拠への接地を必須にする**: 各主張に根拠(実験ID・論文・コード箇所)を付け、
  かつ「自分の主張が覆る条件と、それを確かめる実験を1つ」を添えさせる。
  これで出力が「感想」から「次にやるべき検証リスト」に変わる。
- **できれば実行者を異種にする**: 役割の一部を別ベンダーのモデルに割る。
  同一モデルから複数サンプリングしても意見分布は均質化する(既知の現象)ため。
- **結論は `STATE/decisions.md` に追記する**。ここまでやって初めて審議がループとして閉じる。
